üìö Introduction to BMAD Method

What is BMAD? The Breakthrough Method of Agile AI-driven Development - a framework that combines AI agents with Agile methodologies to accelerate software development.

Key Concepts for Beginners

ü§ñ AI Agents

Specialized AI personalities that handle different roles in development:

  • Analyst - Market research, brainstorming, project analysis
  • PM - Product Requirements Documents (PRD)
  • UX Expert - UI/UX specifications and design
  • Architect - Technical architecture and system design
  • PO - Product Owner validation and document sharding
  • SM - Scrum Master for story creation
  • Dev - Developer for implementation
  • QA - Quality Assurance and code review

üå± Greenfield Projects

Definition: Building from scratch

  • No existing codebase
  • Complete freedom in tech choices
  • Full planning phase required
  • Examples: New SaaS app, MVP, Prototype

üèóÔ∏è Brownfield Projects

Definition: Enhancing existing systems

  • Working with existing codebase
  • Must respect current architecture
  • Integration and compatibility focus
  • Examples: Adding features, refactoring, upgrades
Two-Phase Approach

Phase 1: Planning (Web UI - Cost Effective)

Use large context windows (Gemini, ChatGPT, Claude) to:

  • Generate comprehensive documents (PRD, Architecture)
  • Leverage multiple agents for brainstorming
  • Create once, use throughout development

Phase 2: Development (IDE - Implementation)

Use IDE with AI capabilities (Cursor, VS Code with Copilot) to:

  • Shard documents into manageable pieces
  • Execute focused SM ‚Üí Dev cycles
  • One story at a time, sequential progress
  • Real-time file operations and testing

üå± Greenfield Project Workflow

Building a new project from scratch - Full planning and development process

üìã Phase 1: Planning (Web UI Recommended)

1 Project Analysis & Brief

Analyst

Optional Steps:

  • Brainstorming session
  • Market research
/analyst brainstorm "I want to build a task management app"
/analyst create-project-brief
Output: project-brief.md

Contains: Problem statement, target users, success metrics, MVP scope

Important: Save the output as docs/project-brief.md in your project folder

2 Product Requirements Document

PM
/pm create-doc prd
Output: prd.md

Contains: Goals, requirements, user stories, epics, technical assumptions

Important: Save as docs/prd.md

3 UI/UX Specification (If UI Required)

UX Expert
/ux-expert create-front-end-spec
Output: front-end-spec.md

Contains: User flows, wireframes, component library, design system

Optional: Generate AI UI prompt for v0 or Lovable
/ux-expert generate-ui-prompt
Important: Save as docs/front-end-spec.md

4 Technical Architecture

Architect

Choose based on project type:


# For full-stack projects
/architect create-doc fullstack-architecture
# For backend-only
/architect create-doc architecture
# For frontend-only
/architect create-doc front-end-architecture
Output: architecture.md or fullstack-architecture.md

Contains: Tech stack, components, database schema, API specs, deployment

Important: Save as docs/architecture.md or docs/fullstack-architecture.md

5 Validation

PO
/po execute-checklist-po
Output: Validation report with pass/fail items

May require returning to previous agents to fix issues

üíª Phase 2: Development (Switch to IDE)
Critical Transition: You MUST switch to your IDE (Cursor, VS Code, etc.) for the development phase. Web UI cannot handle file operations.

6 Document Sharding

PO

In your IDE, shard the documents for manageable development:


# Option A: Using PO agent
@po shard docs/prd.md to prd
@po shard docs/architecture.md to architecture
# Option B: Manual with task
# Drag shard-doc task + docs/prd.md into chat
Output:
  • docs/prd/ folder with sharded PRD sections
  • docs/architecture/ folder with sharded architecture

üèóÔ∏è Brownfield Project Workflow

Enhancing existing projects - Requires careful integration with current systems

ü§î First Decision: Enhancement Scope

The workflow changes based on your enhancement size:

  • Single Story (< 4 hours): Use quick story creation
  • Small Feature (1-3 stories): Use epic creation
  • Major Enhancement: Full workflow with architecture
üìã Phase 1: Planning (Web UI Recommended)

1 Enhancement Classification

Analyst
/analyst "I need to add payment processing to my existing Node.js app"
The analyst will help classify your enhancement and route you to the appropriate path

Path A: Quick Single Story (Simple Changes)

PM
/pm brownfield-create-story
Output: Single story ready for implementation

Skip to Development Cycle

Path B: Small Feature (1-3 Stories)

PM
/pm brownfield-create-epic
Output: Focused epic with 1-3 stories

Skip to Development Cycle

Path C: Major Enhancement (Full Workflow)

2 Existing Project Analysis

Architect
Critical: Upload your entire project to Gemini Web or provide GitHub URL
/architect document-project
Output: brownfield-architecture.md

Contains: Current state analysis, tech debt, integration points, constraints

Important: This captures your ACTUAL system state, not idealized architecture

3 Enhancement PRD

PM
/pm create-doc brownfield-prd
Output: prd.md

Contains: Enhancement scope, compatibility requirements, integration strategy

Important: Save as docs/prd.md

4 Enhancement Architecture

Architect

Only if architectural changes are needed:

/architect create-doc brownfield-architecture
Output: architecture.md

Contains: Integration approach, compatibility measures, rollback strategy

Important: Save as docs/architecture.md

5 Validation

PO
/po execute-checklist-po
Focus: Integration safety, compatibility, rollback procedures
üíª Phase 2: Development (Switch to IDE)

Same as Greenfield - shard documents and proceed to development cycle

üîÑ Development Cycle (IDE Only)

The iterative story implementation process - Same for both Greenfield and Brownfield

Critical Context Management: ALWAYS start a new chat between SM, Dev, and QA work to maintain clean context windows!

1 Story Creation

SM

Start NEW CLEAN CHAT


# For sharded PRD (most common)
@sm *create
# For brownfield without sharding
@sm # Then use brownfield-create-story task
Output: story-1.1.md in docs/stories/

Status: Draft ‚Üí Review and approve ‚Üí Approved

Story Structure:
  • User story (As a... I want... So that...)
  • Acceptance criteria
  • Tasks/Subtasks checklist
  • Dev notes with technical context

2 Story Implementation

Dev

Start NEW CLEAN CHAT

@dev # Agent asks which story to implement
# Provide story file content or path
Dev Actions:
  • Implements all tasks/subtasks
  • Creates/modifies files
  • Writes tests
  • Updates File List in story
  • Marks story as "Review" when complete

3 QA Review (Optional but Recommended)

QA

Start NEW CLEAN CHAT

@qa review-story docs/stories/story-1.1.md
QA Actions:
  • Senior dev code review
  • Fixes small issues directly
  • Leaves checklist for remaining items
  • Updates story status
If QA leaves unchecked items, return to Dev (new chat) to address them

4 Repeat Cycle

Continue the SM ‚Üí Dev ‚Üí QA cycle for each story until all epic stories are complete

Remember: Only 1 story in progress at a time. Complete each story before starting the next!
